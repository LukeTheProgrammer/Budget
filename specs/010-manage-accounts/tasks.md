# Tasks: Manage Accounts

**Input**: Design documents from `/specs/010-manage-accounts/`

**Prerequisites**: plan.md, spec.md, research.md, data-model.md, contracts/accounts.md

**Tests**: NONE. Per project constitution (Principle II — No Automated Tests), no test tasks are generated. Verification is manual via `quickstart.md`.

**Organization**: Tasks are grouped by user story. Stories build on a shared foundation (account list page + supporting backend) but each delivers an independently demonstrable increment.

## Format: `[ID] [P?] [Story] Description`

- **[P]**: Can run in parallel (different files, no dependencies on incomplete tasks)
- All paths are repository-relative.

## Path Conventions

Inertia SPA single repo: backend under `app/`, routes under `routes/`, frontend under `resources/js/`.

## Phase 3: User Story 1 — Create a manual account (Priority: P1) 🎯 MVP

**Goal**: A user can add a manual account from the accounts page.

**Independent Test**: Add an account named "Cash Wallet" → it appears in the list and persists across reload; submitting without a name shows a validation error and creates nothing.

- [X] T008 [P] [US1] Create `app/Http/Requests/Settings/AccountStoreRequest.php`: `authorize()` returns true (auth middleware guards the route); rules — `name` required|string|max:100, `type` nullable|enum(AccountType), `currency` nullable|size:3, `last_four` nullable|digits_between:1,4, `balance` nullable|numeric.
- [X] T009 [US1] Add `store(AccountStoreRequest $request)` to `AccountController`: build the account on `$request->user()->accounts()` (forces ownership), default `currency` to `USD`, convert decimal `balance` → integer `balance_cents` (allow negatives, null when blank), save, `Inertia::flash('toast', ['type'=>'success','message'=>__('Account created.')])`, redirect to `accounts.index`.
- [X] T010 [US1] Register `Route::post('settings/accounts', [AccountController::class, 'store'])->name('accounts.store');` in `routes/settings.php`.
- [X] T011 [US1] Create `resources/js/components/accounts/account-form-dialog.tsx` in **create** mode: shadcn `Dialog` + `Form`/`useForm` posting via the Wayfinder `store` action; fields name (required), type (`Select` from `accountTypes`), currency, last_four, balance; surface inline validation errors. Wire an "Add account" button into `settings-accounts.tsx`.

**Checkpoint**: US1 is fully functional and demonstrable on its own (MVP).

## Phase 4: User Story 2 — Edit an existing account (Priority: P2)

**Goal**: A user can update an account they own; linked accounts allow name-only edits.

**Independent Test**: Change an account's name and balance → list reflects it; clearing the name shows a validation error; a linked account exposes only the name field.

- [X] T012 [P] [US2] Create `app/Http/Requests/Settings/AccountUpdateRequest.php`: `authorize()` calls `$this->user()->can('update', $this->route('account'))`; rules — always `name` required|string|max:100; when the bound account is **manual** also allow `type`/`currency`/`last_four`/`balance` (same rules as store); when **linked**, accept only `name` (ignore/forbid other fields).
- [X] T013 [US2] Add `update(AccountUpdateRequest $request, Account $account)` to `AccountController`: persist `name` for all owned accounts; for manual accounts also update type/currency/last_four and converted `balance_cents`; success toast; redirect to `accounts.index`.
- [X] T014 [US2] Register `Route::patch('settings/accounts/{account}', [AccountController::class, 'update'])->name('accounts.update');` in `routes/settings.php`.
- [X] T015 [US2] Extend `account-form-dialog.tsx` with an **edit** mode (PATCH via Wayfinder `update` action, pre-filled values). For linked accounts (`is_linked`), render only the name field editable and show other fields disabled/read-only. Add an Edit control to each account row in `settings-accounts.tsx`.

**Checkpoint**: US1 + US2 work independently; editing respects manual-vs-linked rules and ownership.

## Phase 5: User Story 3 — Delete an account (Priority: P3)

**Goal**: Deleting a manual account soft-deletes it and hides its transactions, while keeping both in the database.

**Independent Test**: Delete a manual account with transactions after confirming → it disappears from the list and its transactions no longer show in Transactions/Dashboard/Budgets; dismissing the confirmation leaves it unchanged.

- [X] T016 [US3] In `app/Models/Account.php`, register a `deleting` model event (e.g. in `booted()`) that, within a DB transaction, soft-deletes the account's related `transactions` so they are hidden together with the account.
- [X] T017 [US3] Add `whereNull('transactions.deleted_at')` (and `whereNull('accounts.deleted_at')` where accounts are joined) to the raw-join spending scopes in `app/Models/Transaction.php` (the `join('accounts', …)` aggregate scopes) and to the `->from('transactions')` raw subquery in `app/Http/Controllers/BudgetController.php`, so soft-deleted transactions stay hidden in those non-Eloquent reads.
- [X] T018 [US3] Add `destroy(Account $account)` to `AccountController`: `$this->authorize('delete', $account)` (403 for non-owner or linked account), `$account->delete()` (triggers the cascade from T016), success toast, redirect to `accounts.index`.
- [X] T019 [US3] Register `Route::delete('settings/accounts/{account}', [AccountController::class, 'destroy'])->name('accounts.destroy');` in `routes/settings.php`.
- [X] T020 [US3] Create `resources/js/components/accounts/delete-account-dialog.tsx` (confirmation `AlertDialog`) issuing DELETE via the Wayfinder `destroy` action; add a Delete control to manual account rows only (hidden for linked accounts) in `settings-accounts.tsx`.

**Checkpoint**: All three stories function independently and together.

## Dependencies & Execution Order

- **Setup (Phase 1)** → **Foundational (Phase 2)** must complete before any user story.
- **US1 (P1)** depends only on Foundational. This is the MVP.
- **US2 (P2)** depends on Foundational; independent of US1 (reuses the dialog component created in US1 — if delivered before US1, create the dialog within US2 instead).
- **US3 (P3)** depends on Foundational; independent of US1/US2.
- **Polish (Phase 6)** runs after the stories being shipped are complete.

Story order by priority: US1 → US2 → US3.

## Parallel Opportunities

- Phase 1: T001 alone.
- Phase 2: T002, T003, T006 are `[P]` (different files); T004 depends on T002/T003 conceptually but edits a different file — sequence T004→T005, T007 after T004.
- Within US1: T008 `[P]` (new file) can be written alongside T011 scaffolding; T009 needs T008; T010 trivial.
- US2's T012 and US3's T016/T017 touch different files and could be developed in parallel across stories once Foundational is done.

## Implementation Strategy

- **MVP = Phase 1 + Phase 2 + Phase 3 (US1)**: a usable accounts page where users can see and create accounts.
- Layer US2 (edit) and US3 (delete) incrementally; each is independently demonstrable.
- Finish with Phase 6 quality gates before considering the feature done.
